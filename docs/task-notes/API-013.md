## 范围
后台运营端所需的身份与权限基座，复用既有不可变 `app.audit_logs`（由 security 模块 `AuditLog` 实体 + `security.audit()` 提供，不新建审计表）。

## 新增内容（独立模块 `src/admin-rbac/`，仅动 `app.module.ts` 一行注册）
- 迁移 `1783768000000-create-admin-rbac.ts`：
  - `admin_roles(id, name, description, created_at, created_by)`
  - `admin_role_permissions(role_id, permission)` —— 复合主键
  - `admin_users(id, email, role_id, created_at, created_by, disabled_at)`
  - `admin_approval_requests(id, actor_id, action, object_type, object_id, payload_json, state[requested|approved|rejected], requested_by, approved_by, decided_at, reason_code, created_at)`
  - 四眼约束：`approved_by IS NULL OR approved_by <> requested_by`
- `admin-rbac.entities.ts` / `admin-rbac.errors.ts`（错误码本地定义，避免改 packages/contracts）
- `admin-rbac.service.ts`：角色/权限判定、审批队列 CRUD、四眼决策、审计导出（只读 `AuditLog`）
- `admin-session.guard.ts`：可撤销、过期、哈希存储的管理员会话守卫；管理员身份由服务端会话派生。
- `admin-rbac.dto.ts` / `admin-rbac.controller.ts`（路由 `v1/admin/*`）+ `admin-rbac.module.ts`

## 端点
- `GET  v1/admin/me` 当前管理员档案（角色+权限）
- `GET  v1/admin/approvals?state=&limit=` 审批队列
- `POST v1/admin/approvals` 发起审批请求
- `PUT  v1/admin/approvals/:id/decide` 通过（四眼：approverId ≠ requestedBy）
- `PUT  v1/admin/approvals/:id/reject` 驳回（不可重复决策）
- `GET  v1/admin/audit?actorType=&userId=&action=&from=&to=&limit=` 审计日志导出（只读）

## 备注
- 审计**写入**走 `security.audit()`，本模块只做**读取/导出**，不碰不可变 `audit_logs` 表结构。
- 审批的**业务执行**（如真正执行赎回）留待 API-008 接入；本模块只负责"请求-审批-决策"状态机与留痕。
- 管理端实际页面在 ADMIN-001~006（独立 `apps/admin`）。
