## 端点
- `POST v1/data-governance/backup-drills` 发起演练
- `PUT  v1/data-governance/backup-drills/:id/finish` 结束演练（succeeded/failed）
- `GET  v1/data-governance/backup-drills` 演练列表
- `POST v1/data-governance/deletions` 发起删除请求（含保留期 retentionDays，默认 30）
- `PUT  v1/data-governance/deletions/:id/decide` 批准/驳回（已决策状态不可再决策）
- `POST v1/data-governance/deletions/purge-expired` 推进保留期到期的物理清除
- `GET  v1/data-governance/deletions` 删除请求列表（可按 state 过滤）

## 设计备注
- **加密**：复用 `IdentityCrypto`（identity 模块），AES-256-GCM + HMAC + 密钥版本化（encryption_key_version 列已是各表既有约束），DB-007 不重复造加密。
- **删除流程**：本模块只推进治理状态机（requested→approved→purged）与留痕；真实数据擦除由各业务模块订阅 `purged` 事件执行行级物理删除，避免触碰现有表。
- **软删列**：仓库内此前无 `deleted_at` 列；为遵守"不改动现有代码"硬约束，DB-007 不给既有表加列，而是通过独立的 `data_deletion_requests` 登记表实现删除流程审计。
- **备份演练**：仅记录演练元数据与结果（满足"恢复演练留存记录"）；实际备份/恢复执行属运维侧，本模块提供可查询的演练台账。
- 审计**写入**仍走 `security.audit()`；本模块作为管理操作，其行为可由 API-013 审计导出追溯。
