# API-005 钱包适配与资金流程实施备注

**状态：** 部分完成  
**日期：** 2026-07-12  
**设计前置：** 已在开工前完整阅读 `DESIGN.md`。本任务只新增 API、适配器与后端状态，不新增界面或概念图。

## 本轮完成

- 新增 `WalletModule`，所有个人余额、地址、提现和转账接口使用真实 Bearer 会话，不再使用固定 Demo 用户。
- 提供 TRON、Ethereum、Arbitrum 三网络策略，返回最小充值/提现最小单位、费用、确认数、预计时间、网络风险提示和配置来源。
- 建立可替换的 `CustodyAdapter` 接口，覆盖钱包/地址创建、地址风险筛查与提现广播形态。
- 内置 `StubCustodyAdapter` 只生成确定性的演示地址并硬性拒绝广播；仅设置 `WALLET_EXECUTION_ENABLED=true` 无法开启真实资金。
- 钱包余额从 `DB-003` 账本账户与余额投影读取，金额全部以 USDT 最小单位整数字符串返回。
- 提现报价验证网络、最小金额、精确手续费和可用余额，不产生资金状态变化。
- 提现与内部转账要求会话绑定的五分钟 step-up 凭证和 `Idempotency-Key`；资金锁定/划转与不可变审计在同一数据库事务中写入双式账本。
- 充值回调使用事件 ID、五分钟时间窗、排序后的规范 JSON 和 HMAC-SHA256 验签；无密钥、坏签名和过期重放均拒绝。
- 同一链交易/输出的重复回调会推进既有 `detected → confirming → credited/manual_review` 状态，不重复入账；HTTP 5xx 明确要求供应方重试。
- 未达到最小金额、地址风险异常或真实执行关闭的充值进入人工审核，不会伪装为已到账。
