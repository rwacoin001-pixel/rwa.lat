# RWA.LAT 多智能体协作开发计划

**状态：** 执行规范  
**产品真相来源：** `docs/RWA-LAT-Product-and-Development-Spec.md`  
**当前设计口径：** 底部为首页、投资、投资组合、钱包；AI 为中间独立按钮；个人中心从右上头像进入。

## 1. 总体判断

当前流程方向正确：先核对原型与文档，再补齐页面和后端，最后接真实合作方。需要增加的关键环节是“契约先行、纵向闭环、独立评审和统一集成”。不要让多个智能体同时在同一个大组件中自由修改，也不要等所有前端页面完成后才开始 API 和数据模型。

建议采用一个指挥者、两个实施角色、一个固定质量门的方式：

```text
产品负责人（最终业务决策）
└─ Codex（总协调、任务拆分、规格维护、集成与最终验收）
   ├─ Claude Code（后端、数据模型、接口、集成测试）
   ├─ Reasonix（独立分析、边界用例、测试设计、代码审查）
   └─ 其他智能体（按任务包承接前端页面、文案、测试或专项工作）
```

Reasonix 若更擅长实现而非审查，也可以承接前端任务，但同一任务的“实现者”和“最终审查者”必须分开。

## 2. 单一事实来源

| 内容 | 唯一来源 | 修改规则 |
|---|---|---|
| 产品范围、页面、用户故事、业务状态 | 产品与开发规范 | 需求变化先改文档，再改代码 |
| 视觉与交互口径 | `rwa-lat/DESIGN.md` 与设计契约 | 与主规格冲突时先提交决策，不自行选择 |
| API 与数据结构 | `packages/contracts` | 前后端不得各自复制类型 |
| 当前任务与负责人 | 本文档的任务表或后续任务文件 | 每个任务只有一个主负责人 |
| 已完成定义 | 自动化检查结果与验收记录 | 只有截图或口头说明不算完成 |

每项需求使用固定编号：`EPIC-模块`、`US-用户故事`、`TASK-任务`、`BUG-缺陷`。提交信息和交接说明必须带编号。

## 3. 推荐实施顺序

```text
冻结导航与页面清单
        ↓
拆分真实前端路由与应用外壳
        ↓
建立共享契约、模拟 API 和状态模型
        ↓
身份/KYC → 钱包/账本 → 投资/订单 → 组合 → AI
        ↓
后台运营、客服、审计与对账
        ↓
合作方沙箱、全链路演练、分地区灰度
```

采用“纵向闭环”交付：每次完成一个能被用户走通的业务片段，同时包含页面、接口、数据、权限、异常状态、测试和审计。例如先完成“登录 → KYC 状态 → 受限页”，再开始“充值 → 到账 → 账本记录”。

## 5. 智能体任务包模板

每次只向一个智能体发送一个边界明确、可在短周期验收的任务包：

```text
任务编号：TASK-XXX
目标：一句话描述用户可获得的结果
依据：规格文档章节、用户故事编号、设计文件
允许修改：明确目录和文件
禁止修改：不属于本任务的模块、视觉口径、业务规则
接口契约：请求、响应、状态、错误码、幂等要求
必须覆盖：默认/加载/空/离线/过期/受限/错误状态
验收用例：正常路径、边界路径、权限路径、失败路径
交付物：代码、测试、截图或接口样例、变更说明
验证命令：可重复运行的检查命令
未决事项：不能自行假设的产品或合作方决策
```

## 6. 开发与交接规则

1. 一个任务只有一个主实现者；其他智能体只做审查或明确分配的子任务。
2. 每个智能体使用独立分支或独立工作树，分支名使用 `agent/任务编号-简述`。
3. 开工前先读取产品规范、设计规范和相关代码；发现冲突先报告，不在代码中偷偷选择口径。
4. 每个任务必须在开工前阅读 `DESIGN.md`，并在交接记录中说明所参考的概念图；没有概念图的重大新页面由主实现者主动生成、审查并纳入原型目录。
5. 不接受首屏、静态截图或空壳功能作为交付。页面必须按正式 App 的完整任务流实现，包含有意义的内容、交互、权限、历史/回执与异常状态。
6. 当真实合作方 API 尚未接通时，先实现类型化适配器和高保真模拟场景；不得用虚构的接口结构或死按钮掩盖缺失能力。
7. 接口先进入共享契约，再分别实现前端和后端；状态枚举、金额、时间与错误码不得重复定义。
8. 交接必须列出修改文件、概念图/设计参考、测试结果、剩余风险和未解决问题，不接受只说“已完成”。
9. 集成由 Codex 统一执行：检查范围、运行测试、核对页面、处理冲突并更新任务状态。
10. Reasonix 优先审查业务边界、异常路径、权限和测试缺口；不要只做代码风格检查。
11. 资金、身份认证、订单和权限代码至少经过一个非实现智能体复核。

## 7. 完成定义

任务只有同时满足以下条件才能关闭：

- 用户故事和验收条件全部映射到实现或测试。
- 已阅读 `DESIGN.md`，并已记录概念图参考或新生成的概念图路径。
- 页面包含完整、可操作的任务内容；没有无意义占位、死按钮、空状态替代真实功能或只完成首屏的交付。
- 不只有理想状态，规定的异常和受限状态均可复现。
- 类型检查、静态检查、单元/集成测试和相关端到端测试通过。
- 前端经过移动端与桌面端视觉检查，无溢出、遮挡或不可点击区域。
- API 包含认证、授权、输入校验、幂等、审计和稳定错误格式。
- 金额与状态可追溯到订单、账本或外部证据，不以客户端计算作为资金事实。
- 文档、契约和代码一致，未决业务问题被显式记录。
- 审查者确认，Codex 完成最终集成验收。

## 9. 仍需产品负责人决策

以下事项会改变实现或合规边界，应由产品负责人明确，智能体不得代替决定：

- 首发国家/地区及每类资产的可用性矩阵。
- KYC、MPC 托管、链上风控、算力、现实世界资产和股票合作方；Polymarket 的代销/分销协议、接口权限与技术 SLA。
- 每类产品的最低额、费率、期限、退出、损失、收益和争议规则。
- 首发 MVP 是否同时开放四类真实投资，还是先以一类资产完成生产闭环。
- 法币入金、会员订阅和邀请奖励是否进入首发范围。
