**版本：** v1.0  
**状态：** 产品基线，待法务、资产供应与合作方确认后进入开发

## 1. 产品定义

RWA.LAT 是以 USDT 为统一资金入口的 AI 驱动全球数字资产投资应用。用户可通过平台托管/MPC 钱包充值、持有、投资与提现 USDT；AI 帮助用户理解组合、风险与投资机会；投资出口覆盖 AI 算力、RWA 项目、美股资产与预测市场。预测市场直接接入 Polymarket 的市场数据与交易接口，RWA.LAT 作为面向用户的代销/分销渠道和订单入口参与业务。

### 1.1 用户核心闭环

`注册 → KYC → USDT 充值 → AI 分析 → 用户确认交易 → 真实结算与持仓 → 收益/市场结算 → USDT 账户 → 再投资或提现`

### 1.2 MVP 的业务边界

MVP 必须支持真实 USDT 资金与真实结算；但不得在未接入合格资产供应、托管、交易及合规合作方前，以“模拟收益”替代真实金融产品对用户出售。开发环境、演示环境和生产环境必须严格隔离。

MVP 资产类别：

1. AI Compute：购买/租赁算力份额，按已披露的产品条款结算收益。
2. RWA：展示、认购、持仓、收益分配与赎回（仅上架已完成发行、资格与披露审核的项目）。
3. Global Stocks：经持牌券商或合规资产合作方提供的交易/持仓能力；不得自行伪造“股票成交”。
4. 预测市场情报：直接通过 Polymarket 接口提供市场浏览、AI 研判、下单、持仓与结算；RWA.LAT 参与代销/分销，不自建独立预测市场，也不以内部模拟订单冒充 Polymarket 成交。生产交易在合作协议、接口权限、地域控制和法律意见齐备后开放。

### 1.3 核心产品原则

- **USDT first：** 所有产品价格、可用余额、收益和手续费以 USDT 为主显示，并以 USD 作为参考值。
- **AI explains, user confirms：** AI 可以提出配置和订单建议；每笔真实交易均须由用户在交易确认页明确确认。
- **Real asset, real disclosure：** 资产来源、最低投资额、费用、预期收益、流动性、期限、风险和结算规则必须可审计、可查看。
- **Custody abstraction：** 钱包供应商必须可替换；业务账本不得以单一供应商 API 返回值作为真相来源。
- **Compliance by design：** 未通过资格、KYC、地域或风险规则的用户不能进入真实交易路径。
- **Crypto-native experience：** 深色、高密度、实时反馈、明确交易状态，但不使用误导性收益、倒计时或风险弱化设计。

## 2. 目标用户与价值主张

### 2.1 首发用户

具备 USDT、熟悉 Web3 钱包与数字资产、希望获得算力/RWA/全球市场机会的全球用户。虽然市场定位为全球，实际可服务范围必须由每个合作方、牌照、制裁与风险政策共同决定。

### 2.2 价值主张

| 用户痛点 | RWA.LAT 解决方案 |
|---|---|
| USDT 分散在多个钱包和平台 | 一个 MPC 托管钱包与统一资产驾驶舱 |
| 不懂如何选择算力、RWA 与预测市场 | AI 解释机会、风险、组合影响与理由 |
| 投资流程复杂、资金流不透明 | USDT 计价、订单状态、持仓和结算流水可追溯 |
| 缺少跨资产的风险视角 | AI Portfolio Score、集中度和情景风险提示 |

### 2.3 商业模式

所有费率必须由后台配置、支持按产品/地区/用户等级覆盖，并在交易确认前展示。

- AI Compute 认购手续费
- AI Compute 管理费与收益分成（如适用）
- RWA 认购/赎回/管理费（按项目条款）
- Polymarket 预测市场代销/分销手续费或合作分成（以正式合作协议和用户披露为准）
- Pro 会员订阅：高级 AI 分析、预警和报告；会员不得绕过合规或风险规则
- 推荐奖励：仅对通过风控和满足归因条件的有效用户/交易发放，后台可配置上限与锁定期

### 3.5 单页面标准工作拆分

每个页面都按同一顺序完成，任何智能体不得只交付理想状态的静态界面：

1. 明确用户、入口、前置资格、主任务、退出路径和敏感数据范围。
2. 定义页面路由、参数、深链接、返回行为和登录/资格守卫。
3. 定义接口契约、数据字段、金额精度、枚举状态、时间戳、错误码和埋点。
4. 完成默认、加载、空数据、离线、过期、受限和错误状态；资金页再完成处理中、完成、拒绝和人工审核状态。
5. 实现响应式布局、触控反馈、键盘与读屏支持、隐私遮罩和国际化，不在组件内硬编码业务文案。
6. 接入模拟接口并完成组件测试；再接入开发环境 API，验证超时、重试、幂等和重复点击。
7. 完成端到端用例、视觉回归、可访问性、性能和真机检查。
8. 由非实现智能体进行代码审查和需求对照；通过后才能标记为完成。

## 4. 用户端产品规范

### 4.1 注册、认证与开户

**支持方式**

- 邮箱 + 密码/验证码
- Google OAuth
- X OAuth
- 外部钱包签名登录（WalletConnect/EIP-4361 风格签名消息；仅用于认证，不等同于平台托管钱包）

**必经流程**

1. 接受服务条款、隐私政策、风险披露与营销授权偏好。
2. 创建或绑定用户账户。
3. 生成/关联 MPC 托管钱包，但不得向前端暴露私钥或助记词。
4. 首次浏览可以不做 KYC；**第一次真实投资前必须完成 KYC 和产品资格校验**。
5. 用户完成 KYC 后才可提交真实订单；提现是否可在 KYC 前进行由合规策略配置，但生产默认应要求 KYC。

**验收条件**

- 同一邮箱、OAuth 身份和钱包地址不可意外创建互相独立的可投资账户。
- 钱包签名只用于当前域名、当前 nonce、短时有效的登录挑战，不可复用为转账或交易授权。
- 登录、KYC 与可投资资格都有独立状态，前端不得仅凭 UI 判断。

#### 首次使用与 Guest 模式

首次启动采用五步、可跳过的引导：

1. **语言与地区提示：** 用户选择语言；展示服务可用性可能取决于资格与所在地，不能把语言选择误作地域资格。
2. **价值介绍：** 用三张简短卡片说明「USDT Wallet」「AI-guided investing」「Global asset opportunities」，不展示未证实的收益承诺。
3. **登录/浏览：** 提供邮箱、Google、X、钱包签名登录，以及 `Explore as guest`。Guest 可查看公开市场信息与产品教育内容，但不能查看个人资产、生成托管钱包、充值、获取个性化 AI 建议或下单。
4. **钱包与安全说明：** 已登录用户创建/关联 MPC 托管钱包；提示 2FA、生物识别和提现保护的用途。
5. **KYC 价值门槛：** 明确说明 KYC 并非浏览前置条件，但首次真实投资前必须完成。用户可从任何投资入口进入 KYC，完成后返回原订单草稿。

首次使用引导的完成状态按步骤分别记录；用户可在个人中心重新查看，不得因跳过营销介绍而绕过安全或 KYC 状态。

### 4.2 首页：资产驾驶舱

**目标：** 用户在 10 秒内理解资产、收益、风险与下一步行动。

**页面模块**

1. 钱包状态：已连接、KYC 状态、USDT 可用余额与快捷充值。
2. Total Portfolio：总资产、24h P/L、累计收益、资产分布。
3. AI Portfolio Score：0–100 分与可解释的主要影响因素；不得暗示保本或收益保证。
4. AI Market Brief：按用户语言生成，带来源时间、数据范围、免责声明和“查看依据”。
5. Opportunities：AI Compute、RWA、Stocks、Prediction 四张资产入口卡；根据资格与地区显示可交易/仅浏览。
6. Recent Activity：充值、订单、收益、结算、提现的最新状态。

### 4.3 投资：资产市场

#### 4.3.1 统一市场规则

- 资产列表至少支持搜索、筛选、风险等级、期限、最低金额、预期收益/概率、流动性和排序。
- 每个资产详情页必须显示供应方、产品文件、价格时间、费用、风险、适当性要求与结算规则。
- 所有“预计 APY”“AI Score”“概率”须标注计算时点、依据与非保证性质。
- 不符合资格的用户可浏览公开资料，但不能提交订单。

#### 4.3.2 AI 算力市场

**列表字段：** 设备型号、区域/供应商、可用率、最小认购额、期限、预期 APY、风险等级、容量状态。

**详情页：** GPU 参数、收益模型、收益来源、预计/历史表现、成本、维护策略、停机政策、提前退出规则、法律与风险文件。

**订单流：**

`选择产品 → 输入 USDT 金额/份额 → 预览费用与预计收益 → 风险确认 → 余额锁定 → 用户二次确认 → 订单受理 → 供应方确认 → 建仓 → 日收益入账`

订单受理不代表成交；若库存、价格或资格变化，订单必须明确失败或部分成交并原路释放未用余额。

#### 4.3.3 现实世界资产投资

**项目类型：** 房地产、能源、基础设施及后续经审核的资产类别。

**详情页必备：** SPV/发行方、资产所在地、法律文件、投资人资格、最小金额、期限、收益分配频率、赎回与转让限制、风险等级、募集进度。

**流程：** USDT 认购 → 资格与文件确认 → 订单受理 → 发行/分配确认 → RWA 持仓凭证 → 按项目规则收益分配/赎回。

#### 4.3.4 全球股票

首发仅在券商/合规资产合作方确认后开放。UI 必须说明标的形式（直接证券、合规代币化凭证或其他）、托管方、报价延迟、订单类型、交易时段、碎股规则、公司行动与汇兑/费用规则。

**禁止：** 在没有执行合作方时将演示行情或内部记账显示为真实股票成交。

#### 4.3.5 预测市场情报 / Polymarket 直接接入与代销

**定位：** RWA.LAT 直接接入 Polymarket 的市场数据和交易接口，并作为面向用户的代销/分销渠道提供发现、分析、下单入口、持仓展示和结算跟踪。用户界面名称统一使用“预测市场情报”。RWA.LAT 不复制一套内部预测市场，不自行决定事件结果，也不把演示行情或内部记账显示为 Polymarket 真实成交。

**业务角色：**

- Polymarket 是预测市场、盘口/报价、成交状态、市场规则和事件结算结果的上游来源。
- RWA.LAT 负责用户界面、AI 辅助分析、用户准入、订单预检、费用披露、代销/分销归因、订单状态同步、用户持仓映射和客服入口。
- 每个 RWA.LAT 预测市场、订单、成交和持仓都必须保存对应的 Polymarket 外部标识，能够双向追踪和对账。
- 正式对外展示 Polymarket 合作或代销身份的名称、商标和措辞，以双方已签署协议及品牌规范为准。

**市场列表字段：** 事件标题、结算时间、YES/NO 概率或报价、成交量、流动性、来源、更新时间、可交易状态。

**详情页：** 事件定义、结算来源、规则、市场深度、走势、AI 观点、相关信号、风险提示、地区/资格限制。

**交易流：**

`选择 YES/NO → 输入 USDT 额度 → RWA.LAT 获取 Polymarket 可执行报价 → 预览份额、价格、费用和最差成交情形 → 阅读 Polymarket 市场规则 → 用户确认 → 资格与余额预检 → 提交 Polymarket 接口 → 同步成交/部分成交/失败 → 映射持仓 → 同步事件结算 → USDT 账本入账`

**接口与异常要求：**

- 报价、市场深度、规则和结算状态均记录 Polymarket 来源标识与更新时间；超过新鲜度阈值时禁止提交订单。
- 订单创建使用平台幂等键，并保存内部订单号与 Polymarket 订单号；重复回调和乱序状态不得造成重复扣款或重复持仓。
- 必须覆盖未成交、部分成交、取消、撤市、无效市场、争议期、结算回滚、接口超时和接口中断。
- Polymarket 接口不可用时，页面保留只读市场与持仓快照，明确最后同步时间，并暂停受影响的报价和下单。
- RWA.LAT 账本记录用户资金事实，但不得覆盖或伪造 Polymarket 的市场成交与结算事实；两者通过对账案件处理差异。

AI 可以给出概率解读、矛盾证据和风险提示，不能以“必胜”“内幕”或确定性语言诱导用户。

### 4.4 AI：投资智能层

#### AI 投资顾问

用户输入资金规模、期限、风险偏好和目标；AI 给出建议配置、资产理由、流动性影响、费用与风险。用户必须能编辑建议；AI 不可直接静默下单。

#### AI 投资组合分析

每日组合分析：资产集中度、收益变化、价格/概率变化、锁定资产、可用余额和风险等级。每条结论应能够展开查看数据来源与时间。

#### AI 机会扫描

扫描已授权的数据源，返回具有解释的算力、RWA、股票和预测市场机会。机会卡不是投资建议或收益承诺，须展示“为什么出现”“主要风险”“数据更新时间”。

#### 一键执行

用户首次使用须单独同意 AI 辅助交易条款。之后 AI 可将用户带入预填订单，但每一笔真实订单仍要求显示订单摘要并由用户确认。不得实现 AI 自主交易、后台自动扣款或跨产品自动调仓。

### 4.5 钱包：托管钱包与资金服务

**支持资产：** MVP 主资产为 USDT；以适配器形式预留 BTC、ETH、RWA Token、Compute Token。

**首发网络：** TRON、Ethereum、Arbitrum。每个网络显示地址、确认数、最低充值额、网络提示、扫描状态和到账时间。

**功能：**

- Deposit：生成/展示用户唯一或归集地址，链上监听确认后入账。
- Withdraw：地址白名单、风险筛查、金额/频率限额、二次验证、审批与链上广播状态。
- Transfer：仅在用户与合规策略允许时开放平台内转账；需显示对方、金额和风险确认。
- Transaction History：充值、提现、内部转账、订单锁定、投资、收益、结算、手续费及调整记录。
- Fiat on-ramp：仅显示“Coming soon”或合规合作方入口；不得在尚无服务时伪造可购买功能。

**余额模型：**

| 余额类型 | 定义 |
|---|---|
| Available | 可用于投资或提现的已结算 USDT |
| Locked | 已被待处理订单或提现占用的 USDT |
| Pending | 链上确认、供应方确认或结算中的金额 |
| Invested | 已转化为产品持仓的成本金额 |
| Reward payable | 已计算但尚未结算为可用 USDT 的收益 |

### 4.6 个人中心：账户、安全与增长

- KYC：状态、资料补充、资格、失败原因和重新提交。
- Security：密码、OAuth 绑定、钱包登录设备、2FA、Passkey、提现地址白名单、会话管理。
- Referral：邀请码、邀请链接、有效邀请数、奖励规则、已发/待发奖励和反作弊状态。
- Subscription：Pro 权益、订阅状态、付款方式、续订和取消；不可用订阅掩盖投资风险。
- Language：七种语言及 RTL 阿拉伯语支持。
- Support & Legal：工单、公告、风险披露、条款、隐私、产品文件与投诉入口。

### 4.7 移动端专属体验规范

#### 生物识别与设备验证

- Flutter Android App 使用 Android BiometricPrompt；设备支持时可启用指纹或人脸认证。PWA 使用平台支持的 Passkey/WebAuthn 作为对应能力。
- 应用不读取、不上传或保存任何生物识别模板；只保存 OS 返回的验证结果及受保护的设备密钥引用。
- 首次启用、重新登录、从后台超过 5 分钟恢复、修改安全设置、创建提现地址、提现，以及单笔超过后台阈值的交易，要求生物识别或 Passkey/2FA 复验。
- 生物识别不可用、失败次数过多或设备变更时，降级为密码 + 2FA；不能以静默方式跳过验证。

#### 深链接

统一优先使用 HTTPS 链接，Android App Links、iOS Universal Links（未来）和 PWA 同时兼容；自定义 scheme 仅用作无法接管 HTTPS 时的回退。

```text
https://rwa.lat/open/asset/{assetId}
https://rwa.lat/open/market/{marketId}
https://rwa.lat/open/portfolio
https://rwa.lat/open/order/{orderId}
https://rwa.lat/open/referral/{referralCode}
https://rwa.lat/open/support/{ticketId}
rwalat://open/...  (fallback only)
```

任何交易、提现或 KYC 深链接只能进入查看/草稿页；必须重新拉取服务端状态并完成当前设备上的用户确认，不能通过 URL 直接执行资金操作。未安装 App 时回退 PWA/网页，未登录时先认证再返回原路径。

#### 推送通知

| 类别 | 事件 | 默认策略 |
|---|---|---|
| Critical security | 新设备、密码/2FA/地址变更、提现请求与完成 | 不可由营销偏好关闭；支持系统级关闭后的应用内提示 |
| Transactional | 充值到账、订单成交/失败、收益到账、赎回、市场结算 | 默认开启；可按子类关闭非关键提醒 |
| Compliance | KYC 需要补件、审核通过/拒绝、账户限制 | 默认开启 |
| Market | 价格提醒、概率变化、AI 信号 | 用户主动订阅，默认关闭 |
| Growth | 邀请、会员、活动 | 用户主动订阅，默认关闭 |

通知载荷不含完整余额、证件信息、地址或交易敏感数据；点击通知只进入经认证的详情页。

#### 更新、加载、离线与空状态

- **PWA：** 新版本在下次启动/刷新时获取；若版本涉及资金流程、安全策略或不兼容 API，阻止交易并要求刷新。静态资源更新不得中断已提交订单状态查询。
- **Flutter：** 普通功能采用商店常规更新；涉及安全、签名、钱包协议或最低 API 兼容性时使用强制更新。强制更新页仍显示系统状态与客服入口，且不吞没待处理订单信息。
- **加载：** 首次加载显示与最终布局一致的骨架屏；按钮仅在当前请求中禁用，并显示可理解的处理中状态。不得用无限 spinner 隐藏合作方超时。
- **离线：** 展示最后成功同步时间与只读缓存；隐藏/禁用充值、提现、确认订单和 KYC 提交，连接恢复后重新校验价格、资格和余额。
- **空状态：** 无持仓显示资产分类与“Explore investments”；无流水提示充值/交易后会出现记录；无可交易产品解释资格、地区或库存原因并提供支持入口。空状态不能伪造盈利或成交数据。

#### 多设备会话

- 允许同一用户最多三个已信任设备并存；新设备登录时向全部已信任设备发送安全通知。
- 用户可在个人中心查看设备、地区/时间、会话状态并逐台或一键注销。密码/双重验证重置默认注销其他会话。
- 两台设备可同时浏览；资金操作以服务端当前余额、nonce、幂等键与风控结果为准，不能依赖本地缓存。
- 新设备在 24 小时内不可新增提现地址；高风险设备或异常地域命中时要求增强验证或冻结资金操作。

### 4.8 通知中心、账户恢复与账户关闭

- 通知中心按 `Security / Transactions / Compliance / Markets / Updates` 分组，支持已读、批量已读和跳转到对应订单/工单；通知内容以服务端状态为准。
- 邮箱账户恢复必须经过一次性挑战、2FA/设备风控和冷静期。钱包签名登录丢失不影响 MPC 托管资产，但解除旧钱包绑定需增强审核。
- 账户关闭前必须处理待处理订单、未结算资产、应付收益、未完成提现、争议工单和订阅。存在受监管记录保留义务时，界面应明确“停止服务”与“立即删除全部数据”的差异。
- 关闭完成后撤销所有会话、OAuth 与外部钱包绑定，冻结新资金动作，并提供带时间戳的关闭确认。

## 5. 资金、订单与结算规范

### 5.1 双式记账账本

PostgreSQL 内部账本是用户余额和产品流水的权威记录。链上交易、MPC 钱包、合作方回报和后台操作均只能通过幂等的记账事件影响余额。

**最低账户分类：** 用户可用资金、用户锁定资金、用户待处理资金、平台清算资金、费用收入、应付收益、资产持仓成本、外部托管对账差异。

**账本不变量：**

- 每一笔记账凭证借贷相等。
- 用户可用余额不允许因并发订单变为负数。
- 链上交易哈希、外部订单 ID、内部请求 ID 和记账凭证 ID 可相互追溯。
- 账本条目不可原地修改或删除；修正必须新增冲正/调整条目并保留原因、操作者和审批记录。

### 5.2 充值状态

`detected → confirming → credited | rejected | manual_review`

充值仅在达到该网络配置确认数、通过地址风险检查并经幂等处理后记入 Available 余额。错误网络、低于最低金额、制裁/高风险命中和异常 token 必须进入隔离或人工审核，绝不自动入账。

### 5.3 提现状态

`requested → 2fa_verified → risk_review → approved → signing → broadcast → confirming → completed | rejected | failed | cancelled`

提现失败时必须根据原因退回 Locked 余额或维持待处理状态；不能简单标记失败却不还款。管理员无法直接修改余额，只能执行需审批的账本调整。

### 5.4 通用投资订单状态

`draft → eligibility_check → quoted → funds_locked → user_confirmed → submitted → accepted → filled | partially_filled | rejected | cancelled → settled`

价格/份额有效期到期、额度不足、KYC 失效、区域限制、合作方拒绝或风控命中都必须返回可解释的失败代码。资金只在 `funds_locked` 后占用；未成交部分在终态自动释放。

### 5.5 收益和结算

- AI Compute：收益计算依据已签署的产品条款、运行数据及收益周期；产出先进入 `Reward payable`，完成对账后转为 Available USDT。
- RWA：按项目分配日、发行方付款和投资人记录执行；赎回按项目规则完成。
- Stocks：以执行/托管合作方的已确认成交和公司行动回报为准。
- Prediction：以产品规则定义的权威结算来源和合作方结算回报为准；争议期、撤市和作废规则必须完整展示。

所有收益计算、费率、舍入方式、结算批次和人工调整均需可导出审计。

## 6. 后台管理与运营规范

### 6.1 管理角色

| 角色 | 最小权限 |
|---|---|
| Support | 查看用户与工单，不可查看敏感证件、不可以资金操作 |
| Compliance | 审核 KYC、资格、可疑交易与地域策略 |
| Treasury | 审批资金归集、提现和对账，不可独自完成高风险动作 |
| Asset Operations | 管理资产目录、供应方、收益批次和产品文件 |
| Finance | 对账、费用、记账调整申请与财务报表 |
| AI Operations | 管理提示词、模型、知识库、评估和成本限额 |
| Admin | 用户/角色管理；不能绕过双人审批与审计 |

### 6.2 后台模块

- 用户与 KYC：状态、风险标签、资格、冻结、申诉。
- 钱包与资金：充值、提现、地址、归集、网络状态、链上对账。
- 资产管理：AI Compute 产品、RWA 项目、股票合作产品、预测市场目录与状态。
- 订单与结算：订单队列、异常订单、收益批次、赎回、争议处理。
- 费率与奖励：按产品、网络、用户等级的可版本化规则。
- Treasury：热/温/冷钱包余额、预留金、资金调拨与日终对账。
- AI：模型供应商、版本、提示词、数据来源、敏感内容规则、成本与人工复核。

### 6.3 四眼原则

以下动作至少要求双人审批，并记录前后差异和理由：高金额提现、地址白名单变更、账本调整、资产上架/下架、收益批次发布、费率变更、风险策略放宽、生产密钥轮换。

### 6.4 客服、投诉与订单争议

MVP 使用原生内嵌 Support Center，而非将用户直接抛到外部网页。Support Center 可通过 SDK 对接 Intercom/Zendesk 等系统，但用户体验和身份映射保留在 App 内。

**客服路径：**

`帮助中心 → AI Support（可选）→ 创建工单 → 补充证据/消息 → 状态追踪 → 解决/申诉`

- AI Support 只回答公开知识库、产品流程和工单引导；不解释个案 KYC 拒绝原因、不作投资建议、不承诺资金处理结果。
- 工单需选择分类、资产/订单/交易关联对象、描述和附件；敏感文件上传至受控对象存储，不进入普通聊天记录或 AI 上下文。
- 订单、提现和结算详情页都有 `Report an issue`；系统自动带入订单 ID、时间、状态和非敏感诊断信息。
- 争议状态统一为 `opened → evidence_requested → under_review → partner_review → proposed_resolution → resolved | rejected → appealed`。每次状态变化产生应用内与合规通知。
- 涉及账户安全、资金丢失、欺诈和错误结算的工单自动提升优先级，冻结/限制动作必须由风控策略和人工授权决定。

## 7. 技术架构

### 7.1 推荐架构

```text
Next.js PWA (mobile-first, i18n, RTL)     Flutter App (Android 12+, Flutter 3.19+)
                     │                              │
                     └──── shared API contracts ────┘
        │
API Gateway / BFF
        │
NestJS modular backend
 ├─ Identity & Access
 ├─ KYC / Eligibility / Risk
 ├─ Wallet Adapter & Chain Indexer
 ├─ Ledger & Reconciliation
 ├─ Asset Catalog & Disclosure
 ├─ Orders & Settlement
 ├─ Portfolio & Pricing
 ├─ AI Orchestration
 ├─ Referral & Subscription
 └─ Admin & Audit
        │
PostgreSQL + Redis + queue/event bus + object storage
        │
MPC Custody | KYC/AML | blockchain RPC/indexer | asset suppliers
Broker/stock partner | Polymarket data/trading API | AI model provider
```

### 7.1.1 客户端与 Monorepo 规范

- **PWA：** Next.js，负责官网、移动 Web、PWA 安装体验和公开 SEO 页面。
- **Android：** Flutter 3.19+，最低 Android 12（API 31），负责生物识别、推送、深链接、设备安全和高频资金操作体验。
- **代码共享原则：** 不强行在 TypeScript 与 Dart 之间共享 UI 或领域代码。共享的真相来源是 OpenAPI/JSON Schema、生成的 API client、错误码词典、i18n message ID、设计 token JSON 和产品配置版本；两端分别生成或实现客户端。
- **推荐 Monorepo：** `apps/web`、`apps/mobile`、`apps/api`、`packages/contracts`、`packages/design-tokens`、`packages/i18n`、`packages/config`、`infra`、`docs`。Flutter 通过 workspace 包或生成物消费 contracts、token 与 i18n；不得复制粘贴错误码或费率展示规则。
- **发布：** Web/PWA 与 Flutter 使用同一 API 版本兼容矩阵。后端废弃字段需至少经历一个移动端发布周期；资金 API 采用显式版本和 server-side capability checks。

### 7.2 必须独立的服务边界

1. **Ledger Service：** 唯一允许写入金融余额与凭证的服务。
2. **Custody Adapter：** 对接 MPC 供应商，屏蔽地址创建、签名、广播、归集差异。
3. **Partner Adapter：** 按资产类型封装报价、订单、成交、持仓、收益和结算回报；预测市场使用独立的 Polymarket Adapter，负责市场同步、订单映射、状态回调/轮询、结算和对账。
4. **Eligibility Service：** 根据 KYC、地域、制裁、产品资格、风险限制决定能否查看/报价/下单/提现。
5. **AI Orchestration：** 接收已允许的数据快照，输出结构化建议与引用；无权触发资金动作。

### 7.3 数据模型（核心实体）

| 域 | 核心实体 |
|---|---|
| 身份 | User, Identity, Session, OAuthAccount, ExternalWallet, Device, Consent |
| 合规 | KycCase, EligibilityProfile, RiskFlag, GeoPolicy, ScreeningCase, AuditLog |
| 钱包 | CustodyWallet, WalletAddress, ChainTransaction, Deposit, Withdrawal, Transfer |
| 账本 | LedgerAccount, JournalEntry, JournalLine, BalanceSnapshot, ReconciliationCase |
| 投资 | Asset, AssetOffering, AssetDocument, Quote, Order, Execution, Position, YieldBatch, Settlement |
| 市场 | PriceTick, Market, MarketOutcome, MarketRule, DataSource, PartnerMarketMapping, PartnerOrderMapping |
| AI | AiConversation, AiRecommendation, AiEvidence, PromptVersion, ModelRun, SafetyReview |
| 商业化 | FeeSchedule, Subscription, ReferralCode, ReferralAttribution, Reward |

所有金额使用整型最小单位或精确 decimal；不得使用浮点数。金额记录同时存储资产符号、网络、精度、汇率来源与时间。

### 7.4 API 规范（资源级）

- `/auth`：注册、OAuth、钱包签名、会话、2FA、Passkey。
- `/me`：用户资料、权限、KYC、资格、设置。
- `/wallets`：托管钱包、地址、余额、充值说明、提现、交易历史。
- `/assets`：资产目录、详情、文件、风险、可交易性、报价。
- `/orders`：创建草稿、报价、确认、取消、状态与回报。
- `/portfolio`：总览、表现、资产分布、持仓、收益。
- `/markets`：Polymarket 市场映射、行情、深度、规则、来源标识、数据时间与可交易状态，以及股票等其他市场数据。
- `/ai`：对话、配置建议、组合报告、机会扫描、引用依据。
- `/referrals` 与 `/subscriptions`：邀请和会员。
- `/admin`：严格 RBAC、审批、配置、审核、对账和审计。

所有资金、订单和后台写请求须接受 `Idempotency-Key`；所有响应包含 `requestId`。外部回调须签名验证、重放保护、顺序无关处理和死信队列。

### 7.5 数据新鲜度、维护模式与降级

- 行情、市场概率、余额、报价与 AI 结论均包含 `asOf`、来源和新鲜度状态。超过产品配置阈值的数据标记为 `stale`，禁止据此提交真实订单。
- 合作方数据中断时保留只读持仓和最后同步时间；隐藏或禁用报价、下单、提现等受影响操作，并显示可追踪的系统状态说明。
- 系统支持全局及按资产/网络/合作方的 kill switch。停机只阻止新动作，不能隐藏既有订单、资金或客服入口。
- 订单提交后即使客户端断网，后台仍持续处理；恢复连接后按订单 ID 拉取权威状态，不得重复创建订单。
- 状态页、维护公告和事件时间线使用同一 incident ID，客服与用户看到一致口径。

### 7.6 可用性、备份与灾备

- 账本、提现、订单状态和 KYC 状态属于 Tier 0；组合展示、AI 摘要和市场筛选属于 Tier 1；营销与内容页属于 Tier 2。
- Tier 0 服务目标：月度可用性 99.95%，恢复时间目标（RTO）≤ 60 分钟，账本与资金事件恢复点目标（RPO）≤ 5 分钟。
- PostgreSQL 使用加密的时间点恢复备份；账本、审计日志和关键配置至少保留一份跨区域副本，并按季度执行恢复演练。
- 队列、RPC、合作方回调和外部数据源均有重试、死信、人工重放与去重策略；恢复期间优先恢复资金状态查询和对账，不自动重放未经审查的资金动作。
- 每次灾备演练生成报告，记录恢复时长、数据差异、失败原因和修复负责人；未通过演练不得扩大真实资金额度。

## 8. 安全、风控与合规发布门槛

### 8.1 技术安全要求

- 不保存或展示用户私钥、助记词、MPC 密钥分片或供应商主密钥。
- 所有服务端机密存入专用密钥管理系统；生产、测试和本地环境物理隔离。
- 用户资金操作使用 2FA/Passkey、设备与会话风险检测、提现延迟/冷静期、地址白名单和额度限制。
- 管理后台使用 SSO、最小权限、硬件/Passkey MFA、IP/设备策略和全量审计。
- 加密货币地址、交易和合作方回调均执行制裁、欺诈和链上风险筛查。
- 定期备份、恢复演练、监控、告警、漏洞管理、渗透测试与智能合约审计（若使用合约）。

### 8.2 合规与法务发布门槛

以下不是可选功能，而是生产启用真实 USDT 和真实结算之前的门槛：

1. 明确主体、牌照/豁免路径与每类产品的发行、经纪、托管、支付与市场执行责任主体。
2. 每个国家/地区的服务可用性规则、制裁筛查和屏蔽策略。`全球` 是获客方向，不是“不做地域控制”。
3. 可验证的 KYC/KYB、AML/CTF、制裁与交易监测流程；KYC 在首次投资前完成是最低产品规则。
4. AI Compute、RWA、股票和预测市场分别具备可签约的资产/执行合作方、真实产品文件、结算 SLA 与争议流程。
5. 用户协议、隐私政策、风险披露、投资者适当性、投诉、数据保留和营销批准流程已由适用法域专业顾问确认。

新加坡的数字支付代币服务处于 MAS 监管框架中，MAS 金融机构目录也列示提供该类服务的持牌机构；MPC/资金服务商筛选应以其具体牌照、产品范围与目标地区为准。[MAS 数字支付代币服务目录](https://eservices.mas.gov.sg/fid/institution?activity=Digital+Payment+Token+Service) 真实美股交易以及事件合约交易还分别需要相应的证券/市场合作结构；例如，美国监管信息将经纪交易与事件合约均列为受监管活动。[SEC 经纪交易商注册指南](https://www.sec.gov/about/divisions-offices/division-trading-markets/division-trading-markets-compliance-guides/guide-broker-dealer-registration) [CFTC 预测市场说明](https://www.cftc.gov/LearnandProtect/PredictionMarkets)

本文件不是法律意见；所有真实资产上线须由持牌合作方和法律顾问书面批准。

### 8.3 隐私与数据生命周期

- 数据按身份、KYC、资金、交易、设备、客服、营销和分析分类，分别定义访问、驻留、保留、导出与删除策略。
- 客户端只采集实现功能所需数据；分析与 AI 上下文遵循最小化、目的限制和可撤回授权。
- 用户可以请求导出个人资料与账户记录；依法必须保留的交易、AML、审计或争议记录不可被普通账户删除流程覆盖。
- KYC 原始文件优先由受审查的 KYC 服务商保管；平台仅保存必要引用、状态和审计证据。
- 日志、崩溃报告、推送与客服系统不得包含认证 token、完整证件、私钥材料或未经掩码的敏感金融字段。

### 8.4 事故响应与紧急控制

| 等级 | 示例 | 处理目标 |
|---|---|---|
| S0 | 账本不平、疑似密钥泄露、未经授权提现 | 立即暂停相关资产动作，15 分钟内建立事故指挥，持续更新状态 |
| S1 | 合作方回报异常、批量订单卡住、链上归集异常 | 30 分钟内限制受影响流程，1 小时内给出临时处理方案 |
| S2 | 非资金页面大面积不可用、AI 数据延迟、推送故障 | 4 小时内恢复或提供降级路径 |
| S3 | 单用户问题、内容错误、低影响体验缺陷 | 纳入客服 SLA 与常规修复 |

- 全局或按资产/网络/合作方的 kill switch 必须具备 RBAC、双人确认、审计日志和自动过期时间；暂停期间仍提供余额、订单、提现状态与客服入口。
- 疑似密钥/设备泄露时，优先撤销会话、冻结高风险提现地址、轮换供应商凭证并启动对账；未经核对不得进行批量补偿或人工改余额。
- 对外状态页、站内通知、推送与客服统一使用 incident ID 和同一事实时间线；事故结束后发布影响范围、资金核对结果和后续措施。

## 9. AI 规范

### 9.1 允许行为

- 使用经批准的行情、链上、产品文件和用户组合数据生成摘要。
- 基于用户授权提供配置建议、风险说明、比较和预填订单。
- 标注时间、来源、局限和冲突信息。

### 9.2 禁止行为

- 承诺收益、虚构数据来源、使用未验证的实时数据。
- 在没有用户逐笔确认的情况下执行订单、提现或转账。
- 使用敏感身份、KYC 文档全文或非必要个人数据作为模型上下文。
- 将 AI Score 当作信用评级、保本等级或适当性判断的唯一依据。

### 9.3 输出结构

AI 建议必须结构化包含：结论、适用的用户输入、数据截止时间、证据/来源、主要风险、费用影响、置信度、免责声明和下一步操作。模型回答无法读取或执行未授权资金动作。

- Logo 采用单一几何符号：一条精确圆环被一条水平纬线切开，右上保留一个微小节点；负形形成简化的 `R`。禁止复杂轨道、多卫星、机器人头像、盾牌和货币符号。

### 10.3 无障碍与平台可用性

- 文本、图标与状态组合达到 WCAG AA 对比度；颜色不是表达状态的唯一方式，涨跌与风险同时使用文字、符号或形状。

## 11. 质量、测试与验收

### 11.1 测试原则

测试外部可观察的资金、订单、权限和展示行为，而不是组件内部实现。最高测试缝隙为“用户/API 请求 → 业务规则 → 账本/合作方回报 → 用户可见状态”。

### 11.2 必测范围

- 身份与权限：所有登录方式、KYC 前后权限、会话失效、2FA 与账户合并。
- 钱包：三链充值确认、重复回调、错误网络、低额充值、提现审批/失败/重试、余额不为负。
- 账本：每个业务流程借贷平衡、幂等、并发下单、冲正、对账差异。
- 订单：报价过期、资格失效、部分成交、合作方超时、取消、结算与用户通知。
- 资产：AI Compute 收益批次、RWA 分配、股票回报、Prediction 的作废/结算争议。
- AI：引用可用性、数据时间、拒绝越权执行、风险提示、不同语言输出与敏感信息保护。
- i18n：七种语言、阿拉伯 RTL、金额/日期格式。
- 安全：API 鉴权、越权访问、重放攻击、回调签名、速率限制、注入与敏感日志扫描。
- E2E：注册 → KYC → 充值 → 真实受控测试订单 → 持仓/结算 → 提现的全链路沙箱演练。

### 11.3 MVP 发布验收

1. 任何余额、订单或收益变化都可从用户页面追溯到不可变账本凭证和外部证据。
2. 任何不符合资格的用户无法绕过 API 或前端提交真实订单。
3. 充值、提现、订单、收益和结算都有可解释终态、通知及失败恢复。
4. 用户在确认页能看到金额、手续费、网络/产品、风险、价格时效和最终确认动作。
5. 关键服务具备监控、告警、备份、灾难恢复演练、权限审计和人工应急停机能力。
6. 每一项真实资产已获得合作方、法务与运营书面上线批准。

### 11.4 分析、可观测性与实验

**工具层建议：** 产品分析采用支持数据区域和隐私控制的 PostHog/Amplitude 类服务；崩溃上报采用 Sentry 类服务；服务端指标、日志和 trace 采用 OpenTelemetry 兼容方案。最终供应商由安全、隐私和数据驻留审查决定。

**统一事件信封：** `eventName`、`eventId`、`occurredAt`、`userId/anonymousId`、`sessionId`、`platform`、`appVersion`、`locale`、`screen`、`requestId`、`experimentVariant`。不得上传私钥、助记词、完整 KYC 资料、完整地址、认证 token 或完整交易金额明细。

**最小事件字典：**

- 获客：`onboarding_started`、`onboarding_step_completed`、`guest_explored`、`signup_completed`、`referral_applied`
- 合规：`kyc_started`、`kyc_submitted`、`kyc_resulted`、`eligibility_blocked`
- 钱包：`deposit_address_viewed`、`deposit_credited`、`withdrawal_requested`、`withdrawal_completed`
- 投资：`asset_viewed`、`quote_requested`、`order_confirmed`、`order_resulted`、`position_viewed`、`settlement_completed`
- AI：`ai_advice_requested`、`ai_recommendation_viewed`、`ai_order_prefilled`、`ai_citation_opened`
- 支持：`support_opened`、`ticket_created`、`dispute_opened`

**性能和可靠性目标：**

| 指标 | MVP 目标 |
|---|---|
| 崩溃/无响应会话率 | ≥ 99.5% crash-free sessions |
| App/PWA 首屏可交互（中位数 / P95） | ≤ 2.5s / ≤ 5s（正常 4G/Wi-Fi） |
| 常规 API 延迟（P95 / P99） | ≤ 500ms / ≤ 1.5s，不含外部交易执行 |
| 订单与资金 API 成功率 | ≥ 99.9%，按幂等重试后统计 |
| 列表滚动帧率 | 目标 60 FPS，P95 不低于 50 FPS |
| 关键后台告警检测 | 5 分钟内发现资金/回调/对账异常 |

实验只能作用于非合规和非资金安全文案、排序或教育内容。KYC、资格、费率计算、交易确认、风险披露、余额和安全控制不允许在未批准的 A/B 测试中改变。

### 11.5 移动端发布与数据质量门槛

- Android 12、13、14、15 及主流屏幕尺寸至少覆盖登录、KYC、充值、提现、订单确认、推送点击、深链接和生物识别回退测试。
- Flutter 发布使用签名密钥托管、依赖锁定、SBOM、静态检查、商店分阶段发布和远程 feature flag；涉及资金协议的版本必须支持强制升级与回滚方案。
- 每次真实资产上线前，必须完成数据源授权、时间戳/延迟检查、缺失值策略、价格/概率异常值检查和人工对账样本。
- 翻译交付包含术语表、上下文截图、RTL 审校、金额/日期格式检查和法律文本人工复核；任何未翻译 key 不得静默显示英文技术 key。
- 生产前执行端到端彩排：KYC → USDT 充值 → 资格 → 报价 → 用户确认 → 合作方回报 → 持仓/结算 → 提现 → 对账 → 客服争议。
- 商店拒审、通知权限关闭、RPC 异常、合作方超时和用户升级失败都必须有可用的降级与人工支持路径。

## 12. 交付计划

多智能体的具体职责、任务包、交接与验收规则见 `AI-Agent-Collaboration-Plan.md`。研发执行遵循以下顺序：先统一规格与契约，再按纵向业务闭环交付，最后接入真实合作方；不得将“全部页面画完”视为可以进入生产集成的条件。

### Phase 0：上线前置（与研发并行，6–10 周或至合作方就绪）

- 主体、牌照/合作路径、地域矩阵与披露审核
- MPC、KYC/AML、链上风险、RPC/indexer、资产/市场执行合作方选型
- 资产上架标准、资金操作流程、对账与事故响应手册
- UX 原型、设计系统与多语言术语表

### Phase 1：平台基础（8–10 周）

- 身份、OAuth、钱包签名、邀请体系、KYC 状态与资格服务
- MPC 钱包适配器、TRON/Ethereum/Arbitrum 充值、提现与双式账本
- 首页、钱包、个人中心、通知、审计后台、管理权限
- i18n 与 RTL 基础设施

### Phase 2：投资与 AI（8–12 周）

- AI Compute 资产、认购、订单、收益与持仓
- RWA 产品目录、文件、认购与收益分配能力
- Polymarket 市场与交易接口、代销归因、规则页、订单映射、持仓同步、结算与对账能力
- AI Advisor、Portfolio AI、机会扫描与逐笔确认

### Phase 3：合作方启用与生产上线（按每类资产批准独立灰度）

- 真实资产合作方沙箱、对账、风控和故障演练
- 分资产、分地区、分额度灰度开放
- 股票合作产品、法币入口、会员订阅与高级分析

生产时间不应以单一“开发完成日”定义，而以每类资产的合作方接入、法务批准、资金对账和安全演练共同决定。
